# ADR-002: 文件分级（L1/L2/L3）控制成本

## 状态

已接受

## 日期

2026-05-22（根据设计文档回溯）

## 背景

在为大量文档构建知识图谱时，对每个文档进行完整的实体提取、关系发现和社区摘要生成会产生大量 LLM 调用，导致成本急剧上升。同时，并非所有文档都具有同等的查询价值——低价值文档（如代码自动生成的日志、格式化输出）往往不需要图谱化。

## 决策

引入文件三级重要性分类，作为成本控制阀门：

| 级别 | 处理方式 | LLM 调用 |
|------|----------|----------|
| **L1** | 仅存储元数据（文件路径、哈希、标题），不做任何 AI 提取 | 0 |
| **L2** | 轻度提取：调用 LightRAG insert，但仅提取实体、跳过社区摘要生成 | 低 |
| **L3** | 完整处理：完整的 LightRAG 实体提取、关系发现、社区摘要 | 高 |

级别由上传时指定（或根据文件特征自动推断），默认 L2。

## 后果

### 正面影响

1. **成本可控**：明确的三级阀门，运维人员可按文件价值分配 LLM 预算
2. **查询质量不降**：高价值文档（L3）享有完整图谱，核心问题不妥协
3. **灵活性**：级别可手动调整，L1 文档后续可升级到 L2/L3

### 负面影响

1. **元数据不可查**：L1 文档无法被 LightRAG 搜索到（Agent 需要知道这一点，必要时使用 `give_up`）
2. **分级判断负担**：需要用户或上传系统判断文件级别，可能误分

### 缓解措施

- 默认级别可配置（`lightrag_importance_level` 配置项）
- 提供批量修改级别的 API，方便调整
- 在系统提示词中告知 Agent：L1 文档不可被语义搜索，需用 SQL 直接查询元数据
