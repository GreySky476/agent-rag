# ADR-001: 选择 LightRAG 作为统一检索后端

## 状态

已接受

## 日期

2026-05-22（根据设计文档回溯）

## 背景

传统 RAG 系统中，图检索（实体关系查询）、向量检索（语义搜索）和社区摘要通常由不同组件维护。例如：
- 向量搜索：使用 Qdrant / Milvus / pgvector
- 图谱查询：使用 Neo4j / Apache AGE
- 摘要存储：另建块摘要向量库

多组件架构带来了以下问题：
- 运维复杂度高：需要部署和维护多个独立服务
- 数据一致性问题：同一文档的图、向量、摘要分别维护，同步困难
- 开发成本高：需要对多个系统的 API 和查询策略

## 决策

使用 **LightRAG** 作为统一检索后端，放弃独立的向量数据库和图数据库组件。

LightRAG 将实体关系提取、社区摘要生成、语义搜索和混合检索整合在一个框架内，并将存储后端全部指向同一个 PostgreSQL 实例（通过 pgvector + AGE 扩展）。

## 后果

### 正面影响

1. **运维简化**：只需维护一个 PostgreSQL 实例，无需部署 Qdrant、Milvus、Neo4j 等额外服务
2. **数据一致性**：LightRAG 内部统一管理实体、关系和向量，避免跨组件同步问题
3. **开发效率**：统一查询接口（`local` / `global` / `hybrid` 模式），减少适配代码
4. **可切换性**：若未来向量规模超出 PG 能力，可单独将 vector_storage 切换为 QdrantVectorStorage，无需改动 Agent 代码

### 负面影响

1. **性能上限**：PostgreSQL 的向量搜索在大规模场景下（>10M 向量）不如专用向量数据库。对策：保留可切换后端的选项
2. **框架锁定**：对 LightRAG 有框架依赖，若其停止维护需迁移
3. **AGE 扩展运维**：PostgreSQL 的 AGE 扩展成熟度不如 Neo4j，需关注稳定性

### 缓解措施

- 配置文件明确各存储后端类型，切换只需改配置
- 定期评估 LightRAG 的社区活跃度和 pgvector 性能
- 保持 Core 层的 `lightrag_tool.py` 接口足够抽象，以便未来替换
